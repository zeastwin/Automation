# Automation 流程编写

## 目标

把用户的业务目标转化为贴合现场、可保存、可继续完善且证据明确的 Automation 流程。方法由本 Skill 维护；字段契约、运行语义、当前资源和平台状态始终从本轮 MCP 的 Schema、Guide 与结构化返回读取。

构造 ChangeSet 前先核对 `automation.md` 中的《ChangeSet V2 构造检查清单》；复杂控制流骨架和边界演算必须通过 `get_process_design_faq` 读取对应主题，禁止在未读 FAQ 的情况下直接构造 ChangeSet。禁止先创建空流程或空步骤再逐步填充。

## 0. 强制读取流程设计 FAQ

加载本 Skill 后、构造首个 ChangeSet 前，必须调用 `get_process_design_faq` 读取与当前用户请求匹配的主题。当前支持的主题：

- `loop`：循环、计数、重复执行、循环 N 次、for 循环、循环流程。

主题不明确或请求涉及多个主题时，先不传 `topics` 读取完整 FAQ，再按主题选取骨架。FAQ 只作为流程编写的参考输入，不替代当前 MCP Schema 和 Bridge 返回的结构化事实。

## 1. 建立本阶段设计卡

在内部整理以下信息，信息已经由用户、当前配置或工具结果唯一确定时直接采用：

- 本阶段要实现的可观察行为和明确不在本阶段处理的范围。
- 输入条件、正常完成、条件分支、失败或超时出口、终止语义。
- 会影响结构的设备、变量、IO、通讯、工站、报警和子流程依赖。
- 本阶段完成后需要哪一级证据：已提交、结构有效、可运行或已实际观测运行。

业务细节尚未明确但不影响骨架保存时，先构造可独立审查的流程或步骤骨架；会改变控制流、安全结果或外部副作用的细节保持未决，取得精确事实后再提交相关指令。

## 2. 只装载当前决策需要的事实

- 复用会话中已经验证且仍适用的对象身份、Schema、资源事实和成功调用经验。
- 已知名称或稳定 ID 时精确读取对象；目标未知时才使用目录或搜索能力。
- 创建、重构或评审复杂流程，且涉及机械反馈、完整控制流、子流程事务、通讯重试、异常恢复或自定义函数边界时，按实际主题调用 `get_process_design_guide`。简单赋值、固定延时、单 IO 操作和字段级编辑直接读取对应 Schema。
- 只有方案依赖某类资源的现有配置或候选名称时，才读取该资源；已经掌握精确名称时可以让预演校验引用。

`ProcessDesignGuide` 提供通用流程设计知识，不提供当前项目资源事实；当前对象和资源由读取工具提供；历史项目案例属于建议性证据，不能替代当前契约、资源状态和安全边界。

## 3. 选择指令表达层

1. 普通业务意图能够由某个语义 `kind` 准确表达时选择语义层。
2. 已选 `kind` 的字段或行为细节会影响结果时，使用 `get_semantic_operation_schema` 精确读取这一种 Schema。
3. 需要保留已知原生 `operaType` 字段，或语义层不能无损表达控制流、安全结构或外部副作用时，选择 `native.operation`，并用 `get_native_operation_schemas` 读取精确原生 Schema。
4. 字段名、枚举、条件必填、数量和引用类型以返回的 Schema 为准；运行联动和失败条件以同源 behavior/Guide 为准。

每条指令按当前目标重新判断表达层。源对象中明确存在的值属于重建约束；源对象没有提供且事实不能唯一确定的值保持未决。

## 4. 组织可审查的 ChangeSet 阶段

- 每个 ChangeSet 只纳入本阶段已经明确的结构、变量声明和动作，可以独立预演、保存并继续完善。
- 现有对象使用稳定 `procId/stepId/opId`；局部 `key` 只连接当前 ChangeSet 内的新对象。
- 重建或精确复制既有步骤时，保留来源顺序和已验证字段，并让预演核对预期的原生指令类型。
- 复杂控制流在首次预演前逐条核对成功后继、失败后继、超时边界和流程终止语义。
- 计数、重试和上下限在首次预演前演算 `N-1`、`N`、`N+1` 三个边界，明确自增发生在判断前还是判断后，并确认第 `N` 次失败进入继续、重试还是超时出口。
- `saveRequired` 决定当前阶段能否保存；只缺 `runRequired` 或晚绑定运行资源的结构可以先保存，启动条件留给 Readiness 判断。

## 5. 预演、确认与提交

1. 使用 `preview_change_set` 预演当前完整阶段。
2. 把返回的 `status`、`confirmed`、`allowedTransitions`、警告和阻塞项视为当前平台事实。
3. 需要修正时，以当前已保存配置为基础重新构造该阶段的完整修正版；旧预演中的局部 `key` 不作为已提交对象身份。
4. 当前状态允许提交且前台已经确认后，仅以该预演的 `previewId` 调用 `apply_change_set`。
5. 提交后从 `createdObjects/affectedProcesses` 取得真实稳定 ID，后续读取、验证和下一阶段编辑均从这些身份开始。

`preview_only` 和 `plannedProcIndex` 描述预演对象，不表示平台中已经存在可读取对象。一次提交成功只证明当前阶段已写入。

## 6. 按用户目标选择验证证据

- 提交结果：证明本阶段已经写入。
- `validate_proc`：证明流程结构满足当前结构契约。
- Readiness 返回：证明当前是否具备启动条件，并列出仍需补齐的运行条件。
- 用户明确要求有界测试时使用 `run_proc_test`；用户明确要求持续运行时才使用持续运行能力。
- 涉及设备、人员或流程安全的不确定状态，保持受影响流程停止并依据平台事实报警。

报告时分别陈述“已提交、结构有效、可运行、测试中已观测运行、自然完成”中的实际证据，不把较低层证据提升为更高层结论。

## 7. 保持事实边界

- 本 Skill：流程编写的决策顺序和验证方法。
- `automation.md`：跨任务稳定的上下文路由入口与 ChangeSet 必填检查清单。
- `ProcessDesignGuide`：通用现场流程设计知识。
- `ProcessDesignFaq`：常见控制流骨架的正例、反例与边界演算（按需读取）。
- MCP 工具描述与参数 Schema：何时调用、输入结构和字段约束。
- Bridge、Store、Readiness 与运行工具返回：当前配置、合法状态迁移、资源状态和运行事实。

同一事实由其权威层维护。发现冲突时以更接近确定性实现的当前结构化事实为准，并回到对应来源读取，而不是在 Skill 中补写一份近似契约。
