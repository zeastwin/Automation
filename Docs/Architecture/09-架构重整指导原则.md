# 架构重整指导原则

本文总结 Automation 本轮大范围架构调整形成的决策原则。它不是维护步骤手册，也不要求为了形式统一继续改造；用途是在以后新增功能、修复缺陷或再次评估技术债时，帮助维护者判断什么值得改、应该改到哪里、何时停止。

## 1. 重整服务于可理解和可验证的行为

架构调整的目标不是换框架、减少行数或增加项目数量，而是让维护者能够回答：

- 程序从哪里启动，实例由谁拥有；
- 配置从哪里读取，由谁提交；
- 流程如何进入运行、暂停、停止和关闭；
- UI、运行内核、设备、Bridge 和 MCP 之间如何传递事实；
- 失败后哪些状态已经改变，哪些仍保持原样；
- 一次改动应通过什么证据证明没有破坏现有行为。

如果一个改造不能减少错误、删除重复事实、明确所有权或建立自动验证，就不应仅以“架构更漂亮”为理由实施。

## 2. 先确认当前事实，再改变边界

重整前先追踪真实调用链、配置路径、线程边界、生命周期和异常处理，不根据类名或目录猜测职责。文档只描述已经存在的事实，目标状态进入技术债或工作包，不能提前写成现状。

一次架构工作包至少要明确：

1. 当前权威事实源；
2. 所有实际调用者和写入入口；
3. 必须保持的正常、异常和安全行为；
4. 需要删除的旧入口、旧字段和重复实现；
5. 可重复执行的验收方式。

## 3. 同一事实只有一个权威来源和一个正式写入边界

同一配置、状态或契约不能由多个模块分别维护近似副本。查询可以投影，UI 可以展示，Bridge 可以序列化，但最终判断和写入必须回到同一个权威来源。

本项目中的典型结果包括：

- `PlatformRuntime` 拥有单个平台实例，替代静态 `SF` 全局定位；
- Store 拥有配置内存事实，窗体和 Bridge 不直接写配置文件；
- `ProcessDefinitionRepository`、`ProcessWorkDirectoryTransaction` 和 `IProcessRuntimeControl` 分别表达编辑态定义、目录事务和运行控制，不再用近义 Store 混合三种概念；
- ChangeSet V2 是 AI 修改流程结构的唯一公开写入链，旧路由和旧字段直接删除；
- MCP 工具集合、参数 Schema、指令定义、行为、readiness 和运行语义各自回到明确权威源码。

发现两条可写路径时，优先删除旁路并让调用者汇入正式边界，不在两条路径之间增加同步代码。

## 4. 依赖从组合根流向具体能力

宿主负责创建并显式传递实例，业务模块接收完成职责所需的最窄对象。非 UI 模块不能通过静态窗体、隐藏窗体或全局服务定位器取得运行能力。

当前依赖方向是：

```text
Program / Host
    -> PlatformRuntime / Composer / Initializer
        -> Store / Engine / Device / Safety / Application Service
            -> 无 UI 契约

Editor / HMI
    -> EditorWorkspace 或 IAutomationPlatform
        -> 上述平台能力
```

物理程序集只用于保护稳定且有实际复用价值的边界。`Automation.Runtime.Contracts` 值得独立，是因为它必须同时服务运行内核和 WinForms 适配器且不能反向引用实现；其他模块在当前规模下使用目录、显式依赖和架构门禁即可，不为了层次感继续拆项目。

## 5. UI 保留 UI 职责，稳定的单页逻辑允许留在页面

窗体可以负责控件创建、布局、事件接收、用户确认和渲染。以下内容才应移出窗体：

- 配置事务、回滚和正式状态替换；
- 跨页面共享的选择或编辑状态；
- 设备协议、运行闸门和安全判断；
- 可独立测试且会被多个入口复用的业务规则；
- 已经导致重复分支或难以验证的提交逻辑。

稳定、只服务一个页面的显示状态和事件逻辑不需要强行抽象。大窗体不是自动成立的技术债；只要剩余内容以布局、转发和渲染为主，就可以保留。

## 6. 保存、运行和安全是三个不同层次

配置可保存不代表流程可运行。保存边界检查结构和可确定不变量；`ProcessReadinessService` 与启动闸门检查运行条件；安全协调器处理设备、人员和状态不确定性。

- 配置缺少运行资源时，可以保存为 `incomplete`，但不能启动；
- 已有配置格式错误或违反约束时，按定义严格失败，不静默猜测；
- 初始化异常应尽量让 HMI 可用，同时停止受影响流程、报警并设置运行闸门；
- 设备或人员安全受到影响时，停止和报警优先于界面完整性或继续执行。

任何重整都不能通过放宽解析、强制结束旧流程、跳过 readiness 或吞掉安全异常来制造“更稳定”的表象。

## 7. 跨文件和跨状态修改必须原子、可恢复、可重复判断

流程与变量、预演与提交、运行时发布与关闭都存在阶段性状态。设计时必须明确提交点、幂等边界和恢复证据。

- 正式状态只在持久化成功后替换；
- 多文件修改使用事务目录、清单和最终提交标记；
- 预演冻结编译结果和基础状态哈希，确认不等于提交；
- `apply_change_set` 只携带 `previewId`，提交前再次校验基线；
- 重复关闭、重复提交和重复启动必须得到确定结果，不能产生第二次副作用；
- 回滚失败必须升级为明确故障和安全锁，不能继续假装状态一致。

“幂等”在这里表示：同一结束动作被重复请求时，系统保持已经完成的结果，不重复释放资源、重复提交配置或破坏当前状态。

## 8. 平台开发阶段不保留没有真实用户的影子兼容层

契约变化时同步更新 Protocol、Bridge、MCP、编译器、前台和测试，并删除旧字段、旧路由、旧工具、旧说明与旧临时脚本。没有真实客户项目依赖的兼容分支只会形成第二条事实链。

删除前先确认全仓调用和外部边界；确认退役后，使用架构门禁防止旧名称和旧入口回流。退役词可以留在反向检查列表中，但不能继续作为可调用能力存在。

## 9. 测试保护行为，架构门禁保护方向

重整不是以“能够编译”作为完成标准。测试分为两类：

- 行为回归：验证事务、readiness、流程生命周期、Bridge 状态机、桌面交互和无编辑器宿主；
- 架构门禁：阻止 `SF.*`、`Application.DoEvents`、旧 Bridge 路由、UI 反向引用、Store 弹窗和已退役类型重新出现。

所有自动回归统一从 `Automation.Core.Tests` 和以下命令进入：

```powershell
dotnet test Automation.sln -c Debug -p:Platform=x64
```

高风险边界同时验证 Debug 和 Release。仍需要真实消息循环、独立进程或完整宿主的场景可以保留隔离测试，但不能重新散落为无人知道是否执行的脚本。

## 10. 注释解释所有权、边界和排查入口

代码注释不逐行翻译实现，而应说明：

- 本文件负责什么，不负责什么；
- 谁拥有当前状态；
- 正式提交点、线程切换点或生命周期入口在哪里；
- 异常时应该沿哪条调用链排查；
- 容易被误删或误合并的设计原因。

字段枚举、工具清单和易变化参数不复制到多个注释或文档中，直接链接权威实现。注释必须帮助维护者理解“为什么在这里”，而不是增加另一份会漂移的说明。

## 11. 避免过度抽象和过度拆分

以下做法通常没有收益：

- 只包装一行调用的 helper；
- 只有一个实现、没有替换需求的透传接口；
- 把一个页面的稳定逻辑拆成多层 Service、Coordinator 和 Adapter；
- 只按行数拆 partial、项目或程序集；
- 为了目录对称给很小且内聚的模块增加层级；
- 新增一层却保留原旁路，导致调用关系更长但所有权不变。

抽取后的对象必须真正拥有一项职责、状态或不变量，并让调用链、测试或复用至少有一项明显改善。否则应留在当前调用处。

## 12. 每个工作包都要删除旧实现并有明确终点

推荐节奏：

1. 读取事实和调用图；
2. 先建立相关回归或架构门禁；
3. 只改变一个所有权或写入边界；
4. 迁移全部调用者；
5. 删除旧代码、旧入口和未使用引用；
6. 执行 Debug/Release 验证；
7. 回读调用图和文档，确认没有形成新旁路；
8. 达到验收条件后停止，不顺手扩大范围。

## 13. 停止与重新启动条件

满足以下条件时，应停止架构重整并回到正常产品开发：

- 主要状态有明确所有者和正式写入边界；
- 启动、配置、运行、安全停止和关闭链能够沿源码读通；
- UI、内核、设备和接入层的依赖方向受门禁保护；
- 关键正常与失败行为可通过统一测试重复验证；
- 剩余问题主要是局部 UI、稳定单页逻辑或没有实测收益的物理拆分。

只有出现以下事实时才重新发起架构工作包：

- 同一事实再次出现两个可写入口；
- 出现绕过事务、readiness 或安全闸门的路径；
- 一个局部修改长期造成三个以上无关模块联动；
- 当前测试无法隔离真实高频故障；
- 构建、部署、复用或框架生命周期产生了当前边界无法解决的实测问题。

行数较多、还能再分类、理论上可以再抽接口，都不是重新开工的充分理由。

## 改动前快速检查

开始一次架构性修改前，先回答：

- 这次要减少哪一种实际错误或重复？
- 当前事实源和正式写入入口分别是什么？
- 改动后的状态由谁拥有？
- 是否会新增一行透传层或第二份近似契约？
- 失败时正式配置、内存状态和设备状态分别如何处理？
- 哪个自动测试或门禁能够证明结果？
- 旧实现能否在同一批次彻底删除？
- 达到什么条件后立即停止？

其中任何关键问题没有答案时，先继续调查，不先增加新抽象。
