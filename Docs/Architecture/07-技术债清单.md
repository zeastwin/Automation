# 技术债清单

## 使用方式

本文只记录有源码证据、能定义验收条件的债务。优先级含义：

- `P1`：下一轮应优先收敛，继续扩展会放大数据一致性、安全或契约风险。
- `P2`：重要的可读性和可测试性问题，应在相关模块再次扩展前处理。
- `P3`：机械性清理，适合并入相邻改造，不单独制造大变更。

截至 2026-07-21 本轮收敛后的规模基线：`AutomationBridgeService.cs` 主文件约 5,316 行，ChangeSet、Migration、Diagnostics、ProcessInspection 已成为独立模块；`FrmMain.cs` 约 1,647 行，`FrmAiAssistant.cs` 约 5,488 行，`InspectorView.cs` 约 4,436 行。数字只用于确定拆分优先级，不作为代码质量结论。

| D-009 | P2 | 当前关键回归主要由 PowerShell 脚本和整库构建承担，实例服务、配置事务、运行闸门和 Bridge handler 的细粒度测试不足。 | 重整只能靠大范围人工回归，导致不敢删除旧代码。 | 建立可在 CI 运行的单元/契约测试项目；覆盖 Store 提交回滚、readiness、Engine 无 UI 运行、ChangeSet 状态机和安全停机。 |
| D-010 | P2 | 主程序仍是一个旧式显式 `Compile Include` 的 .NET Framework 4.8 WinForms 大程序集；逻辑边界大多只靠目录和约定。 | 编译依赖无法阻止反向引用，文件移动成本高。 | 先用依赖测试证明逻辑边界，再按稳定边界抽项目；主程序只引用内核/基础设施，不允许内核反向引用 WinForms。此项不要求先迁移 WPF 或现代 .NET。 |
| D-011 | P3 | `ProcessDefinitionStore`、`ProcessConfigStore`、`ProcessEngineStore` 名称接近但语义分别是内存、目录事务和运行控制。 | 新维护者容易选错入口，进一步制造旁路。 | 在完成持久化收敛后统一命名为 DefinitionRepository/WorkDirectoryTransaction/ProcessRuntimeControl 等清晰概念，或保留名称但增加编译边界和调用规范。 |
| D-012 | P2 | `FrmMain.ShutdownPlatform` 在 UI 线程执行多段同步等待，虽然已有超时保护，仍包含轴任务、流程、通讯和引擎释放的串行协调。 | 某个驱动或后台任务行为异常时，关闭可卡顿且难定位。 | 引入可观测的分阶段关闭协调器；每阶段独立超时和结构化结果；安全停止先完成，UI 可显示当前阶段；重复关闭幂等。 |
