# 重整路线图

目标不是追求某个新框架，而是让系统逐步具备单一事实源、明确依赖方向、可删除的旧代码和可重复验证的行为。每个阶段都必须能独立构建、回归和回滚。

## 阶段一：建立事实基线与人类导航

状态：已完成并由文档链接检查和架构门禁持续维护。

交付物：

- 仓库入口 README；
- 当前系统全景和源码入口索引；
- 启动、配置、流程、编辑器、AI/Bridge、运动安全文档；
- 有证据、优先级和验收条件的技术债清单；
- 文档链接、架构门禁和构建验证结果。

退出条件：新维护者可以在 30 分钟内找到启动入口、流程启动闸门、一次配置提交、AI 写入状态机和安全停机入口；文档不把目标架构写成当前事实。

## 阶段二：收敛 AI 写入契约

目标：让源码只剩一条当前公开写入链。

状态：旧 intent/patch/create_batch 链、模板和客户端展示残留已经删除；ChangeSet V2 是唯一写入链。ChangeSet、Migration、Diagnostics 和 ProcessInspection handler 已从 Bridge 主文件拆出，资源与基础查询模块继续归入 `D-007`，更细的契约测试归入 `D-009`。

工作包：

1. 建立退役路由和客户端方法的精确引用清单。
2. 从 Bridge、模板和说明中删除 intent/patch/create_batch 等旧写入链；Profile 中仅保留退役词作为反向门禁。
3. 保留 `preview_change_set -> 确认 -> apply_change_set(previewId)`。
4. 将 ChangeSet 路由、预演仓储和提交服务从 9 千行 Bridge 文件中抽出，但不改变协议。
5. 增加 Profile、Schema、状态机和提交结果契约测试。

退出条件：全仓退役写入词为零；Editor Profile 和 Bridge 实际路由一致；预演替换、确认、过期、状态哈希冲突、提交和丢弃均有自动测试。

## 阶段三：收敛配置所有权和安全入口

目标：每类配置一个正式提交入口，每类安全停机一个协调入口。

状态：主体代码已完成。窗体和 Bridge 不再直接读写 JSON，Store 不依赖 UI，安全停机统一到 `PlatformSafetyCoordinator`；细粒度事务、回滚和故障注入测试继续归入 `D-009`。

工作包：

1. 先处理 IO 和工站，将窗体/Bridge 的直接 `AtomicJsonFileStore.Save` 移入应用服务或 Store。
2. 为提交定义统一结果：校验错误、磁盘结果、正式内存是否变化、运行时发布、回滚状态、需要刷新哪些 UI。
3. 再处理 IO 调试、卡和剩余配置旁路。
4. 统一 `FrmMain.StopAllProcsForSafety` 与 `PlatformSafetyCoordinator`。
5. 让 Store 返回结构化错误，删除 `CardConfigStore` 的 MessageBox。

退出条件：UI 和 Bridge 不直接写配置文件；提交失败不改变正式状态；跨文件操作有事务；所有安全异常经过同一协调服务并有故障注入测试。

## 阶段四：隔离流程内核与 WinForms

目标：`Engine/` 可以在没有 WinForms 消息循环和窗体的测试进程中运行。

状态：弹窗边界已完成。`IProcessPopupService` 承载结构化请求，WinForms 适配器负责窗口、计时、取消和 UI 调度；Engine 已无 WinForms、具体窗体和 `UiInvoker` 引用。无 UI 行为测试和分阶段关闭协调器仍待完成。

工作包：

1. 定义流程交互端口，表达弹窗请求、确认、取消、超时和流程停止。
2. 将 Popup operation 中的 `Control`、WinForms Timer 和具体 `Message` 窗口移到 WinForms 适配器。
3. 删除 `ProcessEngine.UiInvoker` 和 Engine 下所有 WinForms/VisualStyle 引用。
4. 为暂停、单步、停止、弹窗取消和进程关闭添加无 UI 测试。
5. 抽取分阶段关闭协调器，保留安全停止优先和各阶段超时。

退出条件：Engine 项目级依赖扫描不含 WinForms；核心流程测试无需创建窗口；关闭与弹窗行为保持一致。

## 阶段五：按页面纵向拆分编辑器

目标：窗体只处理布局、事件转发和展示，业务规则进入可测试应用服务。

状态：主窗体横向减负已经完成一轮：设备与 EngineContext 组合进入 `PlatformRuntimeComposer`，配置发布、设备/轴监视和系统状态进入 Runtime 服务，流程弹窗和通用页面宿主进入 `UI/`。HMI 不再创建隐藏 `FrmMain`；下一步转入各页面的纵向拆分，并继续把重复关闭协调收敛为单一生命周期服务。

一次只处理一个页面，建议顺序：

1. `FrmIODebug`
2. `FrmValue`
3. `FrmProc` / `FrmDataGrid`
4. Inspector
5. `FrmAiAssistant`
6. 最后缩小 `FrmMain`

每个页面采用相同节奏：先画状态与依赖图，锁定行为测试，抽提交/查询服务，抽页面状态模型，最后删除窗体内重复分支。不要同时改视觉样式和业务语义。

退出条件：单个窗体不再拥有配置事务、设备协议和跨模块业务规则；事件处理函数能沿明确服务调用链读到底；页面回归可独立执行。

## 阶段六：固化程序集边界与技术升级选择

目标：用编译依赖代替口头约定。

候选边界：

- `Automation.Core`：流程模型、readiness、执行内核和无 UI 端口；
- `Automation.Infrastructure`：JSON 配置、日志、通讯和硬件适配；
- `Automation.WinForms`：平台编辑器和内置 HMI；
- 现有 `Automation.Protocol`、`Automation.DeviceSdk`、`McpServer` 继续保持专用职责。

只有逻辑边界和测试先稳定后才移动项目。到这一阶段再评估从 .NET Framework 4.8 升级现代 .NET；WPF 迁移仍是独立产品决策，不作为偿还核心架构债的前提。

## 每个工作包的固定节奏

1. 列出当前权威事实源、调用者和必须保持的行为。
2. 增加或确认针对本工作包的回归门禁。
3. 只改一条边界，不混入新功能和无关视觉调整。
4. 删除旧入口、旧字段、旧测试和旧说明，不保留影子兼容层。
5. Debug 与 Release 构建，运行相关回归和架构扫描。
6. 回读调用图，确认依赖方向真的改变，而非只新增一层透传包装。
7. 更新本架构导航和技术债状态。

## 暂不做的事情

- 不以 WPF 重写代替内核解耦。
- 不先做大规模命名或文件移动，再寻找边界。
- 不一次拆成几十个项目。
- 不在重整批次中顺便增加产品功能。
- 不为已经退役且没有真实客户契约的路径保留兼容层。
