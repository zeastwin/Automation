# Automation AI 流程设计指南

> 版本：3.0  
> 适用对象：Automation 内置 AI 助手、流程设计人员、流程评审人员  
> 适用结构：`Proc → Step → OperationType`

---

## 1. 文档定位

这份指南只解决一个问题：如何把用户描述的设备行为设计成结构清晰、可保存、可验证、便于恢复的 Automation 流程。

它负责：

- 把自然语言目标转换为可观察的流程结果；
- 划分 `Proc`、`Step` 和 `OperationType` 的职责；
- 设计机械动作、等待反馈、控制流和异常恢复；
- 判断何时拆分子流程或自定义函数；
- 在信息不足时形成可继续完善的流程骨架；
- 检查流程设计是否形成完整闭环。

它不负责：

- 规定变量名称、类型、初始值或槽位；
- 提供固定的 IO、点位、速度、时间和报警编号；
- 复制语义指令或原生指令的字段 Schema；
- 替代 `OperationBehaviorCatalog` 对指令运行行为的定义；
- 替代现场机械、电气和安全评审；
- 规定 ChangeSet 的字段、提交协议或工具调用参数。

当前平台事实按以下来源确定：

- 指令字段：当前语义或原生 Schema；
- 指令行为：当前行为契约和 Guide；
- 资源事实：变量、IO、工站、点位、通讯、PLC、报警等资源工具；
- 可保存性：当前 ChangeSet 编译与保存结果；
- 可运行性：当前 readiness 和运行闸门；
- 实际行为：本轮真实测试或实机观测。

# 第一部分：从行为目标开始

## 2. 先定义可观察结果

流程设计从设备最终应呈现的行为开始，而不是从指令列表开始。

设计前先明确：

1. 流程从哪个稳定状态进入？
2. 用户最终希望看到什么设备行为？
3. 哪些事实可以直接证明目标已经完成？
4. 哪些动作会改变机械、设备或外部系统状态？
5. 哪些失败允许自动恢复，哪些失败需要停止和报警？
6. 流程结束后设备应停留在哪个稳定状态？

例如，“执行一次取料”的目标不能只表达为“打开真空”。完整目标至少包括：

```text
入口：机构处于允许取料的稳定位置
动作：移动到取料位置并建立真空
完成证据：运动完成且取料反馈成立
提交时机：反馈成立后确认本次取料成功
失败结果：恢复到安全状态，或停止并进入明确异常出口
结束状态：机构和本次动作结果均可被后续流程可靠判断
```

## 3. 建立内部设计卡

复杂流程在创建前形成一张内部设计卡。设计卡用于约束方案，不要求逐项展示给用户。

| 项目 | 需要明确的内容 |
|---|---|
| 入口状态 | 允许流程开始的稳定状态和前置条件 |
| 目标行为 | 用户要求设备完成的实际行为 |
| 完成证据 | 能直接证明目标完成的反馈或结果 |
| 机械对象 | 轴、气缸、吸头、输送线、机器人等 |
| 外部对象 | 子流程、通讯设备、PLC或外部系统 |
| 命令动作 | 会产生机械、设备或数据副作用的动作 |
| 反馈信号 | 到位、停止、真空、响应、完成或其他终态 |
| 安全互锁 | 动作开始前必须成立的条件 |
| 等待边界 | 反馈等待、工艺等待或事务等待的结束条件 |
| 异常出口 | 超时、故障、停止、取消和重试耗尽后的去向 |
| 恢复节点 | 恢复成功后允许返回的稳定位置 |
| 重复执行 | 当前动作或事务再次执行是否安全 |
| 未确定项 | 当前证据不能唯一确定的资源或参数 |
| 验证证据 | 结构、就绪、测试或实机层面的完成依据 |

## 4. 只读取影响当前设计的事实

流程设计所需事实由当前动作决定。

### 4.1 IO 动作

关注：

- 精确 IO 资源；
- 输入或输出方向；
- 命令状态和完成反馈；
- 对向或互斥输出；
- 动作前安全条件；
- 等待超时和异常出口。

### 4.2 轴或工站运动

关注：

- 精确工站、轴和目标点；
- 回零、使能和报警状态；
- 目标有效性和运动边界；
- 安全位、工作位和离开路径；
- 与其他轴、气缸或机构的互锁；
- 运动完成的实际反馈。

### 4.3 子流程

关注：

- 被调用流程的精确身份；
- 启动前置条件；
- 启动后的预期终态；
- 等待边界；
- 子流程停止、失败或超时后的处理；
- 重复启动是否安全。

### 4.4 通讯或外部事务

关注：

- 精确设备和协议对象；
- 请求发送和响应完成的判断；
- 协议成功与业务成功的区别；
- 超时、错误分类和允许重试的错误；
- 写操作重复执行的影响；
- 事务失败后的确定出口。

### 4.5 信息不足

当前事实不足以唯一确定运行参数时，流程可以先形成：

- 空流程或步骤骨架；
- 已明确的前置检查和控制结构；
- 已确定的安全动作；
- 保留运行资源缺口的 incomplete 配置。

缺失的 IO、点位、速度、超时、报警或通讯参数保持未决，由当前资源事实补齐后再进入运行就绪状态。

# 第二部分：Automation 流程结构

## 5. Proc 的职责

一个 `Proc` 表达一个可以独立启动、等待、停止、完成或恢复的任务。

适合作为独立流程的任务通常具有以下一种或多种特征：

- 有明确入口和完成终态；
- 可以被主调度流程独立启动和等待；
- 具有独立的机械或外部事务边界；
- 需要独立停止、重试或恢复；
- 被多个流程复用；
- 需要与其他工站并行运行。

简单且只在当前步骤内使用的连续动作，可以保留在同一流程中，不必为了形式完整拆成多个空壳流程。

## 6. Step 的职责

一个 `Step` 表达流程中的一个业务阶段，而不是任意数量的指令容器。

推荐按阶段划分：

```text
00_入口与状态确认
10_安全与前置检查
20_准备目标或参数
30_执行动作
40_等待反馈
50_结果判断
60_完成提交
90_异常收敛
99_结束
```

实际流程只保留有意义的阶段。简单流程可以只有少量步骤，复杂流程也可以按机械节拍或业务事务继续细分。

一个步骤应当具备：

- 清晰的阶段目标；
- 可理解的进入条件；
- 与阶段目标一致的指令；
- 明确的正常出口；
- 必要时具有异常出口。

## 7. OperationType 的职责

一条指令只承担其当前契约定义的动作。

指令名称应表达真实意图，例如：

```text
检查_机构是否允许下降
运动_Z轴到安全位
输出_取料气缸伸出
等待_取料气缸伸出到位
判断_本次动作是否成功
结束_本次取料流程
```

名称用于帮助阅读；实际行为、字段和跳转仍以指令 Schema 和行为契约为准。

## 8. 常见流程层次

### 8.1 初始化流程

初始化流程通常负责：

- 检查设备当前状态；
- 建立通讯和设备可用性；
- 完成必要回零；
- 使执行器进入安全状态；
- 处理断电或中断后遗留的物理状态；
- 建立可启动其他流程的稳定终态。

初始化成功的判据是设备状态已被确认，而不只是初始化命令已经发出。

### 8.2 主调度流程

主调度流程通常负责：

- 检查模式和全局运行条件；
- 启动、等待和协调工站流程；
- 处理节拍关系；
- 响应暂停、停止和退出；
- 汇总工站终态。

具体气缸、真空和轴动作放在承担相应机械任务的工站流程中，使调度层保持清晰。

### 8.3 工站流程

工站流程完成一个明确机械任务，例如进料、取料、搬运、加工、检测、放料或出料。

一个完整工站流程通常具有：

- 稳定入口；
- 前置条件；
- 机械动作闭环；
- 完成终态；
- 超时或故障出口；
- 可返回的稳定恢复节点。

### 8.4 事务流程

复杂通讯、第三方设备交互或需要独立等待的业务动作，可以设计成事务流程。

调用方负责准备本次任务、启动事务、等待确定终态并消费结果。事务流程负责完成一次完整交互并进入确定终态。

### 8.5 恢复流程

恢复流程处理可以被明确、安全恢复的异常，例如：

- 动作失败后的机构退回；
- 通讯中断后的重连与重新执行；
- 满盘或缺料后的人工处理；
- 可重试动作的有限重试；
- 中断后返回安全位置。

恢复成功后返回哪个稳定节点必须明确。无法证明恢复安全时，受影响流程进入停止和报警状态。

# 第三部分：动作闭环

## 9. 通用动作结构

所有具有机械、设备或外部副作用的重要动作采用以下结构：

```text
检查前置条件
→ 发出命令
→ 等待反馈
→ 验证结果
→ 提交完成状态
→ 正常继续或异常收敛
```

其中：

- 前置检查证明当前允许开始动作；
- 命令只代表请求已经发出；
- 反馈证明设备或外部对象发生了预期变化；
- 验证用于排除矛盾信号和关联故障；
- 完成提交发生在物理或业务结果确认之后；
- 异常出口使流程回到可解释状态。

## 10. 命令与完成分离

以下动作都需要区分“命令已接受”和“目标已完成”：

- IO 输出与气缸到位；
- 真空输出与真空建立；
- 运动命令与轴停止、到位；
- 流程启动与流程到达预期终态；
- 通讯发送与响应完成；
- 响应收到与业务成功；
- 复位命令与故障实际清除。

当设备提供完成反馈时，以反馈作为完成证据。固定延时适用于工艺保持、去抖、信号稳定或明确要求的等待，不替代已有的完成反馈。

### 10.1 IO 布尔值与机构语义

IO 的 `true/false` 表示精确点位的运行时逻辑激活状态，不统一表示安全、危险、原位或工作位：

- 输出 IO 为 `true` 表示该输出逻辑激活，为 `false` 表示该输出逻辑未激活；
- 输入 IO 为 `true` 表示该传感器条件成立，为 `false` 表示该传感器条件不成立；
- 电气高低电平及正常/取反属于底层映射，不作为机构位置语义；
- 安全位、工作位、原位和动位由部件类型、目标动作及对应反馈共同确定。

设计 IO 动作时先确定机构目标，再分别确定命令和完成证据：

```text
机构目标
→ 输出 IO 的目标逻辑值
→ 目标位置传感器的期望逻辑值
→ 对向位置传感器的期望逻辑值
→ 完成或异常出口
```

本项目的常用机构语义：

| 对象和目标 | 命令或反馈 |
|---|---|
| 安全门已关闭 | 关门输入为 `true`，表示该安全条件成立 |
| 单电磁阀气缸伸出到工作位 | 输出为 `true`，动位输入为 `true`，原位输入为 `false` |
| 单电磁阀气缸缩回到原位 | 输出为 `false`，原位输入为 `true`，动位输入为 `false` |
| 真空建立吸附 | 真空输出为 `true`，真空检测输入为 `true` |
| 真空释放 | 真空输出为 `false`，真空检测输入为 `false`；破真空动作按实际机构契约处理 |

双线圈气缸、保持型阀、取反 IO 或具有特殊反馈逻辑的机构以当前资源和设备契约为准。现有事实只能确定单个 IO、无法确定输出与原位/动位反馈的配对关系时，配对和目标值保持未决。

## 11. 状态提交时机

计划和物理结果属于不同阶段：

```text
选择目标
→ 校验目标
→ 执行物理动作
→ 确认物理结果
→ 提交业务完成状态
```

例如，找到目标位置只证明已经完成规划；运动、释放和反馈确认完成后，才能把目标视为本周期已完成。

异常发生在提交之前时，后续恢复仍能依据真实物理状态决定重试、撤回或停止。

## 12. 气缸动作

气缸动作通常采用：

```text
检查安全条件和关联机构位置
→ 确认对向输出处于允许状态
→ 输出动作命令
→ 等待目标到位反馈
→ 校验对向反馈和关联故障
→ 确认动作完成
```

双线圈或互斥输出的动作，需要按实际电气契约处理输出顺序。实际输出电平和反馈逻辑由当前 IO 资源与指令契约确定。

## 13. 真空取放

取料闭环：

```text
确认允许取料
→ 建立真空
→ 等待真空反馈
→ 验证取料结果
→ 成功继续，或进入有限重试/异常出口
```

放料闭环：

```text
确认允许放料
→ 释放真空或执行破真空
→ 等待释放反馈或完成条件
→ 验证产品已释放
→ 离开工作位置
→ 提交本次放料完成
```

动作失败后的真空保持、关闭或释放策略由现场安全和产品保护要求决定。

## 14. 轴和工站运动

典型运动结构：

```text
检查回零、使能、报警和目标有效性
→ 检查机构互锁
→ 移动到安全过渡位置
→ 等待运动完成
→ 移动到目标上方或工作位置
→ 等待并验证到位
→ 执行工艺动作
→ 返回安全位置
```

安全过渡位置是否需要、采用哪条路径以及运动顺序，由当前机构结构决定。设计目标是使每一步都具有可验证的安全前提和完成反馈。

动态目标在发出运动前完成范围和有效性校验。计算目标、校验目标和执行运动在流程结构中保持可辨识。

# 第四部分：控制流

## 15. 分支设计

每个业务判断都应形成完整的控制流：

- 条件成立时进入哪个目标；
- 条件不成立时进入哪个目标；
- 哪一条路径属于正常继续；
- 哪一条路径属于异常处理；
- 两条路径最终在哪里汇合或结束。

“继续下一条”也应符合当前指令的实际控制流契约。未知原生指令的顺序和跳转行为通过精确 Guide 确认。

## 16. 等待设计

等待分为三类：

### 16.1 反馈等待

等待设备或外部对象达到明确状态，例如到位、停止、响应完成或流程结束。

设计中明确：

- 等待对象；
- 目标状态；
- 超时边界；
- 超时出口；
- 停止请求的响应方式。

### 16.2 工艺等待

用于保压、固化、稳定或其他由工艺明确要求的持续时间。时间结束本身就是完成条件。

### 16.3 节拍等待

用于流程协调或短暂退让。它不承担设备到位证明，也不替代反馈等待。

## 17. 循环设计

循环必须具有确定出口。常见出口包括：

- 目标已经完成；
- 用户或上层流程请求停止；
- 等待超时；
- 重试次数耗尽；
- 设备进入故障状态；
- 当前任务被取消。

轮询循环每次重新读取当前状态，并为其他停止或调度动作保留响应机会。

## 18. 正常结束

流程自然完成时进入明确结束路径。

结束前确认：

- 本次目标是否已经得到完成证据；
- 机械对象是否处于允许的稳定状态；
- 本次动作是否仍有未结束的等待或子流程；
- 后续流程能否唯一判断本次结果。

正常结束、失败停止、用户取消和异常中止具有不同含义，流程结构应保留这种区别。

## 19. 禁用和跳过

功能未启用或本周期明确跳过时，流程直接走确定的旁路路径：

```text
判断当前是否需要执行
├─ 需要 → 执行完整动作闭环
└─ 不需要 → 进入明确跳过终态
```

跳过路径不执行部分副作用动作，也不伪装成动作成功。安全互锁的旁路需要来自明确授权和现场安全设计。

# 第五部分：子流程与事务

## 20. 子流程调用闭环

主流程调用子流程时采用：

```text
检查子流程可启动条件
→ 准备本次任务
→ 启动子流程
→ 等待子流程进入预期终态
→ 判断子流程结果
→ 正常继续或进入异常出口
```

主流程不把“启动成功”当作“任务完成”。等待应覆盖子流程自然完成、失败、停止和超时等可能终态。

## 21. 并行流程协调

多个流程并行运行时，明确：

- 各流程独立负责的设备对象；
- 共享资源的占用边界；
- 谁发起本周期任务；
- 谁判断整体完成；
- 一个流程失败时其他流程如何停止或收敛；
- 暂停、停止和复位如何传播。

共享机械区域、共享通讯通道和共享产品位置需要确定性的互斥或握手机制。

## 22. 通讯事务闭环

标准通讯事务采用：

```text
检查设备可用性
→ 准备本次请求
→ 清理旧响应或旧事务状态
→ 发送请求
→ 等待响应并处理超时
→ 校验协议结构
→ 校验业务结果
→ 正常完成或进入分类失败
```

发送失败、接收超时、协议错误、业务拒绝和连接中断是不同失败类型。恢复策略以实际错误类型为依据。

## 23. 重试与重连

自动重试满足以下条件：

- 当前错误属于允许重试的类型；
- 重试次数有限；
- 每次尝试有独立等待边界；
- 重试前清理会污染本次判断的旧状态；
- 重复执行不会造成不可接受的副作用；
- 达到上限后进入确定失败出口。

重连只恢复连接可用性。连接恢复后，当前业务事务仍需要重新执行并校验。

写入、上传、下发和数据库事务在重试前确认重复执行语义。无法证明幂等时，流程进入查询确认、人工恢复或确定停止路径。

# 第六部分：异常与恢复

## 24. 异常必须收敛

异常路径最终进入可解释的稳定状态：

- 动作已安全撤回；
- 机构保持在现场规定的安全状态；
- 流程停止并明确报警；
- 等待人工处理；
- 或已经有限恢复并返回指定稳定节点。

异常处理不以“跳过当前指令继续运行”作为默认结果。

## 25. 稳定恢复节点

稳定恢复节点具有以下特征：

- 当前机械位置和反馈可以被确认；
- 后续动作的前置条件可以重新检查；
- 不依赖已经失效的临时判断；
- 重复进入不会产生额外副作用；
- 可以明确继续、重试或停止。

恢复流程应返回稳定节点，而不是返回发生故障的动作中间位置。

## 26. 自动恢复边界

自动恢复适用于结果可验证、风险可控且次数有限的动作。

自动恢复设计明确：

- 恢复动作的安全前提；
- 最大尝试次数；
- 每次失败后的状态清理；
- 恢复成功的直接证据；
- 重试耗尽后的停止和报警出口。

设备、人身、产品或当前物理状态存在不确定性时，受影响动作停止并明确报警。

## 27. 停止、暂停和取消

流程设计需要考虑外部控制请求可能发生在等待和循环期间。

- 暂停保留继续执行所需的可信状态；
- 停止使受影响动作进入平台规定的停止状态；
- 取消终止本次任务并进入确定终态；
- 急停和工业安全动作由平台及现场安全链路处理。

机械动作中途能否暂停或恢复，由实际执行器和运行契约决定。

# 第七部分：自定义函数边界

## 28. 适合放入流程图的动作

以下内容通常保持在流程中可见：

- 安全和前置检查；
- 单次 IO 输出；
- IO 或设备反馈等待；
- 轴和工站运动；
- 流程启动与等待；
- 关键业务判断；
- 超时、报警和恢复路径；
- 正常结束。

这些动作直接影响现场调试和流程控制，应能从流程结构中看到。

## 29. 适合使用自定义函数的能力

以下内容可以由自定义函数承担：

- 复杂协议转换；
- 外部 SDK 事务；
- 复杂目标计算；
- 数据库或文件事务；
- 多流程复用的纯业务算法；
- 多个底层调用必须形成的单一业务事务。

关键安全互锁和需要在流程图中直接观察的机械动作保持显式。

## 30. 自定义函数的流程契约

流程使用自定义函数前，需要掌握：

- 函数目的；
- 输入来源；
- 输出和成功条件；
- IO、文件、数据库、通讯等副作用；
- 最大执行时间；
- 内部是否重试；
- 重复调用是否安全；
- 失败类型；
- 如何响应流程停止；
- 调用完成后流程应进入哪个分支。

方法名称本身不能证明这些运行语义。

# 第八部分：典型流程骨架

## 31. 单气缸动作

```text
00_入口
  确认本次动作目标

10_检查
  检查安全条件
  检查关联机构位置
  检查对向状态

20_执行
  发出气缸动作命令

30_等待
  等待目标到位反馈

40_判断
  到位且状态一致 → 完成
  超时或状态矛盾 → 异常

90_异常
  执行现场允许的安全收敛
  报警或停止

99_结束
```

## 32. 真空取料

```text
00_入口
  确认取料任务和稳定入口

10_检查
  检查产品存在
  检查机构允许取料

20_运动
  到达取料位置
  等待并验证到位

30_取料
  建立真空
  等待真空反馈

40_判断
  反馈成立 → 离开取料位置
  反馈失败 → 有限恢复或异常

50_离开
  返回安全位置
  等待并验证到位

60_完成
  确认本次取料成功

99_结束
```

## 33. 轴运动到工作位

```text
00_入口
  明确目标位置和完成条件

10_检查
  检查回零、使能和报警
  校验目标
  检查机构互锁

20_安全过渡
  到安全位置
  等待并验证完成

30_目标运动
  到工作位置
  等待并验证到位

40_工作动作
  执行当前工艺动作

50_离开
  返回安全位置

90_异常
  停止受影响运动并报警

99_结束
```

## 34. 异步子流程事务

```text
00_入口
  明确本次任务和预期终态

10_准备
  检查子流程状态
  准备本次请求

20_启动
  启动子流程

30_等待
  等待完成、失败、停止或超时

40_判断
  成功终态 → 正常继续
  失败终态 → 异常处理
  超时 → 停止或恢复

99_结束
```

## 35. 通讯请求

```text
00_入口
  明确设备和本次请求

10_检查
  检查连接和设备可用性
  检查请求完整性

20_发送
  清理旧响应
  发送本次请求

30_等待
  等待响应或超时

40_校验
  校验协议结构
  校验业务结果

50_恢复
  对允许重试的错误执行有限重试
  必要时重连后重新执行事务

90_失败
  进入确定失败出口

99_结束
```

# 第九部分：设计检查

## 36. 创建前检查

- [ ] 用户目标已经转换为可观察行为；
- [ ] 稳定入口和稳定结束状态已经明确；
- [ ] 完成证据不是单纯的命令发送成功；
- [ ] 当前动作涉及的资源已经按需读取；
- [ ] 未确定资源和参数保持未决；
- [ ] 机械和外部副作用已经识别；
- [ ] 每个重要动作的前置条件已经明确；
- [ ] 等待对象、完成条件和超时出口已经明确；
- [ ] 自动重试的安全性和重复执行影响已经判断；
- [ ] 异常后的稳定恢复节点已经明确。

## 37. 创建中检查

- [ ] `Proc` 表达独立任务，未形成无意义的空壳拆分；
- [ ] `Step` 按业务阶段划分；
- [ ] 指令名称能说明当前动作目的；
- [ ] 重要动作形成“检查—命令—反馈—验证—提交”闭环；
- [ ] 有设备反馈时没有用固定延时代替反馈；
- [ ] 运动经过当前机构要求的安全路径；
- [ ] 业务完成只在实际结果确认后提交；
- [ ] 判断的所有业务出口完整；
- [ ] 循环具有成功、停止、超时或重试耗尽出口；
- [ ] 子流程调用包含启动、等待和终态判断；
- [ ] 异常路径能够收敛到稳定状态；
- [ ] 当前阶段只包含已有事实能够支持的设计。

## 38. 创建后检查

- [ ] 流程结构验证已经执行；
- [ ] 跳转目标和流程出口有效；
- [ ] readiness 和运行阻塞已经核对；
- [ ] 需要字段级确认的对象已经回读；
- [ ] 保存成功、结构有效和运行就绪被分别判断；
- [ ] 用户明确要求测试时才执行有边界测试；
- [ ] 测试结果只描述本轮实际观测；
- [ ] 未经实机验证的机械行为保留相应结论边界。

# 第十部分：核心设计准则

1. 从可观察目标出发，再选择流程结构和指令。
2. `Proc` 表达独立任务，`Step` 表达业务阶段，指令表达单一契约动作。
3. 复杂度由真实行为决定，不为形式完整制造额外层级。
4. 所有重要动作采用“检查—命令—反馈—验证—提交”结构。
5. 命令接受和动作完成是不同事实。
6. 有反馈时，以反馈证明完成；固定延时服务于明确的时间语义。
7. 机械动作开始前验证安全前提、目标和机构互锁。
8. 复杂运动按当前机构要求经过可验证的安全路径。
9. 计划目标和物理完成分阶段处理。
10. 业务状态只在物理或外部事务结果确认后提交。
11. 分支的所有业务出口明确，循环具有确定退出条件。
12. 子流程启动后等待并判断其真实终态。
13. 通讯区分发送、响应、协议成功和业务成功。
14. 自动重试有限、有独立边界，并以重复执行安全为前提。
15. 重连恢复连接后，当前业务事务仍需重新执行和校验。
16. 异常路径返回稳定恢复节点，或停止受影响动作并明确报警。
17. 基础动作、安全互锁和恢复路径保持在流程中可观察。
18. 复杂算法或外部事务通过具有明确契约的自定义函数承载。
19. 缺失资源保持未决，允许形成 incomplete 骨架。
20. 设计建议不能替代当前 Schema、行为契约、资源事实和运行证据。
